# Как запустить "взрослый" эксперимент: 36 моделей, пошагово

Всё уже собрано и проверено оффлайн. Ваши действия - пять команд в PowerShell.
Папка: `C:\Users\Sergey\archipelago_cross_model\experiments\reliability-full`.

## Шаг 0. Перед запуском (один раз)

1. Откройте `PROTOCOL_FULL.md` и впишите дату запуска в строку
   "Run date locked" - это фиксация пререгистрации.
2. Ключ (если ещё не задан):
   ```
   $env:OPENROUTER_API_KEY = "sk-or-v1-ваш-ключ"
   ```
   Баланс: хватит $40-60 (жёсткий потолок в команде запуска - $60,
   реальная оценка обычно $10-40, точную покажет dry-run).

## Шаг 1. Оффлайн-проверка (бесплатно, ~10 секунд)

```
cd C:\Users\Sergey\archipelago_cross_model\experiments\reliability-full
python run_reliability.py --selftest
python analyze_reliability.py --run _selftest_run --items _selftest_items --panels panels.json --selftest-panels
```
Обе команды должны отработать без ошибок (вторая напечатает метрики на
симулированных данных).

## Шаг 2. Сгенерировать банк заданий (бесплатно)

```
python items_bank.py --seed 20260721 --tier-mix "1:5,2:10,3:15,4:15,5:5" --out items_v1
```
50 задач, эталоны посчитаны кодом, файл с ответами моделям не показывается.

## Шаг 3. План и оценка стоимости (бесплатно)

```
python run_reliability.py --catalog models_catalog.json --items items_v1/items.json --out run_v1 --dry-run
```
Покажет: сколько вызовов (36 моделей x 5 партий = 180), оценку в долларах по
живым ценам и предупредит, если какой-то слаг пропал из каталога (тогда
пришлите мне список - обновим карту слагов, у моделей в config_phase2l.yaml
есть запасные варианты).

## Шаг 4. Запуск (это и есть эксперимент)

```
python run_reliability.py --catalog models_catalog.json --items items_v1/items.json --out run_v1 --max-usd 60
```
Идёт от получаса до пары часов (медленнее всего рассуждающие модели). Обрыв
сети или упор в потолок - не проблема: повторный запуск той же команды
продолжит с места остановки (готовые партии пропускаются). Сырые ответы
каждой модели сохраняются в `run_v1\raw\`.

## Шаг 5. Анализ (бесплатно, локально)

```
python analyze_reliability.py --run run_v1 --items items_v1 --panels panels.json
```
Результат в `run_v1\`:

- `reliability_metrics.json` - точность всех 36 одиночек; точность панелей
  D3/D5 (сырое разнообразие), DQ3/DQ5 (разнообразие без слабых), R3/R5
  (блок консенсуса), ATLAS3, TOP3/TOP5, полоса случайных панелей;
  доверительные интервалы; сигнал сходимости (точность при единогласии
  против точности при расколе) и таблица риск-покрытие.
- `panel_accuracy.png` - главная картинка: столбики стратегий на фоне серой
  полосы случайных панелей.

## Как читать итог (критерии из пререгистрации)

- DQ3 выше серой полосы - разнообразие с контролем силы работает (H1).
- D3 ниже полосы - сырое разнообразие вредит, и это тоже публикуемый
  результат с числом (H2).
- Разрыв "единогласие против раскола" от 20 пунктов - главный практический
  продукт: правило "принимай при единогласии, эскалируй при расколе" (H3).
- Если медиана точности пула выше 90% - сработал потолок: команда из
  протокола (seed 20260722, уровни 2-6) и перезапуск (правило эскалации
  зафиксировано заранее, это не подгонка).
