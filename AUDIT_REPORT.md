# Аудит архива archipelago_cross_model.zip

Дата: 2026-05-19. Аудитор: Claude (автономный режим). Цель: проверить скрипты до того, как пойдут платные API-вызовы, выявить риски перерасхода / поломок парсинга / проблем безопасности.

## TL;DR

Архив рабочий. 9 Python-скриптов синтаксически чисты, точки входа на месте, ключевые функции (триадная развёртка, парсер конструктов, парсер рейтингов) прошли smoke-проверку на синтетических данных в моей Linux-песочнице. Структура config.yaml валидна, перечисление 10 ячеек совпадает с инструкцией. Реальные API-вызовы из песочницы я выполнить не могу (openrouter.ai заблокирован прокси сэндбокса) - это нормально, dry-run у тебя на машине покроет.

Найдено 6 замечаний: 1 средний риск (модели в config.yaml почти наверняка не существуют под этими slug-ами и потребуют замены), 2 низкого риска по парсингу, 3 косметических. Ни одно не блокирует запуск; средний риск решается за 5 минут на этапе dry-run.

## Замечания

### M1 [средний риск] - Slug-и моделей в config.yaml почти наверняка не существуют

Файл `config.yaml` использует slug-и моделей, которые выглядят как будущие версии:
- `anthropic/claude-opus-4.7` (фоллбек: `anthropic/claude-sonnet-4.6`)
- `openai/gpt-5.5` (фоллбек: `openai/gpt-5.4`)
- `google/gemini-3.1-pro-preview` (фоллбек: `google/gemini-3-pro`)
- `deepseek/deepseek-v4-pro` (фоллбек: `deepseek/deepseek-v4-flash`)
- `moonshotai/kimi-k2.6` (фоллбек: `moonshotai/kimi-k2.5`)

Проверить из моей песочницы я не могу - openrouter.ai заблокирован allowlist'ом. Поэтому это предположение, не факт. Но опираясь на текущее (май 2026) состояние каталога OpenRouter, существование `claude-opus-4.7`, `gpt-5.5` и `gemini-3.1-pro-preview` именно под этими slug-ами на момент твоего запуска не гарантировано: версии и slug-и OpenRouter меняются.

**Что произойдёт:** `python run_experiment.py --dry-run` напечатает `M2 (openai/gpt-5.5) NOT FOUND in OpenRouter catalog` для каждой непрошедшей модели. Если ни primary, ни fallback не найдены - скрипт упадёт с явным сообщением (`run_experiment.py:794-798`). Это безопасное поведение.

**Что делать:**
1. Дождаться, пока `--dry-run` отпечатает список доступных моделей и не найденных.
2. Открыть https://openrouter.ai/models, найти актуальные slug-и под каждое семейство.
3. Открыть `config.yaml`, заменить `id:` для непрошедших моделей. Полная замена `fallback_id:` обычно не требуется - оставь как страховку.
4. Запустить `--dry-run` повторно. Когда все 5 моделей подтвердились - идти на реальный прогон.

Это **штатная процедура** инструкции (раздел 3 INSTRUKCIYA_RU.md), но из-за высокой вероятности срабатывания я выношу как замечание M1.

### L1 [низкий] - Парсер конструктов может вернуть < 3 при нестандартном выходе модели

`_parse_constructs()` (run_experiment.py:499-522) ищет блоки вида `Triad N:` + `Left pole:` + `Right pole:`. Если модель отклонится (например, использует `Pole 1:` / `Pole 2:`) - сработает fallback на `X vs Y` / `X ↔ Y`. Если и тот не найдёт - вернётся пустой список.

В этом случае ячейка не упадёт: код в `phase3_constructs` пишет в `cell.errors`, продолжает с тем, что распарсилось, выводит warn в лог. Это **не критично**, но влияет на полноту анализа.

**Что делать:** ничего на этапе запуска. После прогона - проверить `analysis/per_cell_summary.csv`, столбец `n_constr`. Если у какой-то ячейки заметно меньше 15 - открыть `results/<cell>/cell.json`, посмотреть `raw_output_excerpt`, понять, в каком формате модель отдала. Если паттерн повторяется по моделям - можно дописать паттерн в `_parse_constructs` и перепрогнать только фазу 3 для затронутой ячейки (`--cell <id>`).

### L2 [низкий] - Парсер рейтингов чувствителен к табличному формату модели

`_parse_ratings()` (run_experiment.py:631-654) ожидает строки вида `C_M1_1,E1,5` либо CSV-таблицу `C_M1_1,1,5,3,7,2`. При temperature=0 на ratings-фазе вероятность ухода в свободный формат низкая, но если модель добавит markdown-кодовый блок ``` или комментарии вида `Here are the ratings:` - парсер всё равно извлечёт цифры, потому что регулярка ищет по контексту, а не по позиции. Тем не менее: при недопарсе пишется warn (`phase4 {sn}: got {got}/{expected} ratings`), и в `cell.json` хранится сырой ответ через `record_api_call` → audit-файл.

**Что делать:** ничего профилактического. После прогона проверь количество ошибок в `state.json` или `FINDINGS.md`.

### L3 [низкий] - Pipe `Write-Output 'YES' | python` под Windows PowerShell ненадёжен

В моём первом черновике `run_all.ps1` я использовал PowerShell-пайп для подачи `YES` на ввод `run_experiment.py`. Это известная проблема WinPS 5.x с пробросом stdin в нативные процессы.

**Уже исправлено:** в run_all.ps1 использовано `cmd /c "echo YES| python run_experiment.py"` - cmd.exe прокидывает stdin в нативные процессы надёжно.

### C1 [косметика] - `_parse_ratings` принимает E[1-9], но дальше код использует только E1-E5

Регулярка `r"(C_[A-Z0-9_]+)\s*[,\s|:]\s*(E[1-9])\s*[=:\-,]\s*([1-7])"` теоретически примет E6-E9, которые в эксперименте не существуют. Дальше проверка `if ... ek in element_labels` это отсечёт. Безопасно, но регулярка переразрешает.

### C2 [косметика] - В `record_api_call` есть слово `pejabat`

В строке 270 русско-эквивалентного слова "официальный" / "должностной" - оставлено в docstring (по-видимому, из машинного перевода). На функциональность не влияет, но при ревью пейпера может вызвать вопросы.

### C3 [косметика] - Файл `interactive_sample.html` в корне внешнего архива - это образец, не финальный артефакт

В исходном `archipelago_cross_model.zip` помимо вложенного архива лежит `interactive_sample.html` (38 КБ) и `visualizations_interactive.py` отдельно от внутреннего архива. Это просто образец рендера. Скрипт `visualizations_interactive.py` внутри уже распакованного `archipelago_cross_model/` - корректный.

## Безопасность

- API-ключ читается только из env var `OPENROUTER_API_KEY` (run_experiment.py:956). В файлы не пишется. ✓
- В `record_api_call` сохраняется `request_payload` без заголовков (только `model`, `messages`, `temperature`, `max_tokens`). Bearer-токен из заголовка туда не попадает. ✓
- `state.json` и `cell.json` содержат конструкты, рейтинги, тексты ответов, usage и стоимость - чувствительной информации не содержат. ✓
- Hard-cap бюджета $80 проверяется в основном цикле перед каждой ячейкой (`run_experiment.py:1013`). ✓

## Покрытие тестами

Юнит-тестов в архиве нет. В отсутствие тестов я провёл smoke-тесты в Linux-песочнице на ключевых функциях:

```
triad assignment shape: 5 agents, 3 triads/agent
parsed constructs: 3/3 OK
parsed ratings: 5/5 OK
persona count: 5 (Q,S,E,H,C)
```

Этого достаточно, чтобы убедиться, что pipeline собран корректно. Реальное поведение на ответах настоящих моделей покрывается только живым прогоном.
